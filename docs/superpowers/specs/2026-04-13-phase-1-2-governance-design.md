# 2026-04-13 首轮治理方案设计（Phase 1-2）

## 背景

当前项目已经具备较完整的业务能力，但工程层面存在明显的第二阶段治理需求：仓库边界、运行时目录、配置加载、启动流程、环境模式、规划文档状态之间存在不一致与耦合。这些问题会直接抬高后续后端分层重构、前端 feature 化重构以及部署治理的成本。

本设计文档不覆盖全量系统重构，而是聚焦首轮治理：**Phase 1 仓库边界与目录治理** 和 **Phase 2 配置、启动与运行模式治理**。目标是在不改变核心业务行为的前提下，为后续彻底治理建立稳定地基。

## 目标

### Phase 1 目标
- 将源码、运行时数据、构建产物、部署产物从职责上彻底分开
- 收敛 `.gitignore`，去除重复、历史遗留和边界不清规则
- 建立统一目录职责说明，使仓库具备长期维护能力

### Phase 2 目标
- 明确系统启动流程：配置加载、数据库连接、迁移、HTTP 启动的职责边界
- 建立 development / test / production 的配置策略
- 降低因默认值、隐式路径、环境差异造成的运行风险

## 非目标
- 不在本轮引入大规模后端业务分层重构
- 不调整前端 feature 结构
- 不修改核心业务接口语义
- 不在本轮完成备份、迁移、审计、安全体系的全面重构

## 现状判断

### 仓库与目录边界问题
当前仓库中混入了运行产物和大体积本地依赖，说明源码仓库承担了过多运行时职责。根目录 `.gitignore` 规则重复较多，前后端规则混杂，维护成本高，也容易出现“本地环境干净但仓库语义不干净”的问题。

### 启动与配置问题
后端 `main.go` 当前承担加载配置、连接数据库、自动迁移、启动 HTTP 服务等多个职责。`config` 层已经具备基础校验，但仍偏向本地开发导向，生产环境边界不够清晰。路径配置、运行目录、默认值策略尚未形成统一规则。

### 项目状态一致性问题
规划文档与实际里程碑存在漂移，说明过程管理和真实代码状态未形成单一可信来源。虽然这不阻碍系统运行，但会影响后续治理判断与团队协作。

## 方案概览

本轮采用“先边界、后启动”的治理顺序：

1. **先治理仓库和目录职责**，避免后续结构调整继续被运行时杂质干扰
2. **再治理配置与启动流程**，为后续后端分层和部署治理提供稳定入口

这是一个低业务扰动、但高长期收益的首轮治理方案。

# Phase 1：仓库边界与目录治理设计

## 设计原则
- 源码仓库只保存可版本化资产
- 运行时数据不进入版本控制语义
- 目录职责必须能通过名称和文档快速理解
- 不为了短期方便继续容忍“运行数据贴着源码存放但语义不清”的状态

## 目标目录职责
建议将项目目录职责明确为：

- `backend/`：后端源码
- `frontend/`：前端源码
- `docs/`：项目文档与设计文档
- `deploy/`：源码级部署配置（若确认保留）
- `uploads/`：运行时上传目录，不纳入版本控制
- `backups/`：运行时备份目录，不纳入版本控制
- `logs/`：运行日志目录，不纳入版本控制
- 临时目录、构建产物目录、可执行文件目录：全部视为运行时产物，不纳入版本控制

如果 `deploy/` 包含的是可版本化部署脚本、配置模板和容器编排文件，则保留；如果是本地打包产物或一次性交付文件，则应迁出源码职责。

## `.gitignore` 设计
本轮应将根目录 `.gitignore` 收敛为简洁、职责清晰的规则集合，按以下板块组织：

1. 依赖与包管理缓存
2. 前端构建产物
3. 后端二进制与测试产物
4. 环境变量与本地配置
5. 日志与临时文件
6. 运行时目录（uploads/backups/logs）
7. IDE / OS 文件
8. 本地工作目录（如 `.worktrees/`、`.agents/`）

原则上：
- 重复规则删除
- 过时规则删除
- 仅保留当前项目真正使用或高概率产生的规则
- 前后端通用规则放根目录；局部规则保留在子模块 `.gitignore`

## 文档设计
新增一份目录职责文档，说明：
- 哪些目录属于源码
- 哪些属于运行时
- 哪些目录可以删除后由系统自动重建
- 开发、测试、部署三种场景下目录如何使用

## Phase 1 交付物
- 精简后的根目录 `.gitignore`
- 必要时更新 `backend/.gitignore` 和 `frontend/.gitignore`
- `docs/project-structure.md`
- 统一后的目录职责说明

## Phase 1 风险与缓解
### 风险 1：清理规则时误伤仍依赖的本地文件
缓解：先列出现有未提交文件与运行目录，确认哪些必须保留为本地文件，哪些只是历史残留。

### 风险 2：某些脚本隐式依赖旧目录
缓解：在修改规则前，检查启动、部署、备份相关脚本中的硬编码路径。

### 风险 3：团队成员对 `deploy/` 职责理解不一致
缓解：先定义“源码级部署配置”和“交付产物”的边界，再决定保留或迁出。

## Phase 1 验收标准
- 克隆仓库后不会自动携带运行时垃圾
- `node_modules`、备份 zip、上传文件、日志、二进制不再进入版本控制语义
- 新成员仅靠目录结构与文档即可理解仓库职责

# Phase 2：配置、启动与运行模式治理设计

## 设计原则
- 启动入口只做装配，不承载具体业务初始化逻辑
- 配置必须显式、分域、可校验
- 开发环境的便利不能污染生产环境的安全与稳定边界
- 运行路径必须配置化，不依赖隐式相对路径猜测

## 启动流程设计
建议将后端启动流程拆成以下职责：

1. **Load config**
   - 加载环境变量
   - 执行配置校验
2. **Build infrastructure**
   - 建立数据库连接
   - 初始化文件存储相关目录（若必须）
3. **Run migration policy**
   - 按受控策略决定是否自动迁移
4. **Build HTTP app**
   - 组装 router、middleware、handlers
5. **Start server**
   - 启动 HTTP 服务

`main.go` 应只保留上述装配逻辑，不直接堆积实现细节。

## 配置结构设计
建议将配置拆成明确的结构域：

- `App`
  - 环境名
  - 监听端口
  - 前端静态目录
- `Database`
  - 主机、端口、用户名、密码、数据库名
- `Auth`
  - JWT Secret
  - 默认密码策略
- `Storage`
  - 上传根目录
  - 文件下载目录（如需）
- `Backup`
  - 备份根目录
- `CORS`
  - 允许来源列表

## 环境模式策略
### development
- 允许从 `.env` 加载
- 可保留部分安全默认值以简化本地开发，但应尽量减少
- 自动迁移策略可开放，但必须显式可见

### test
- 使用独立测试配置
- 不依赖开发环境本地运行目录
- 尽量使用隔离数据库和隔离运行路径

### production
- 禁止关键项依赖默认值
- 必须显式提供敏感配置
- 运行目录必须由配置指定
- 自动迁移是否开启应作为明确策略，而非默认行为

## `.env` 策略设计
- 增加 `.env.example` 作为唯一模板来源
- 明确 `.env` 仅用于本地开发或受控环境
- 生产环境使用真实环境变量或安全配置注入方式
- 错误信息应明确指出缺失的是哪个配置项、属于哪个配置域

## 路径治理设计
当前上传目录、备份目录、前端 dist 目录存在相对路径使用倾向。本轮应将这些路径纳入配置，形成统一规则：

- 应用代码不假设工作目录
- 运行时路径通过配置注入
- 本地开发与部署环境可使用不同路径，但结构语义一致

## 自动迁移策略设计
当前自动迁移直接放在应用主启动流程中。本轮不强制完全移除，但需要先设计清楚策略：

- development：允许自动迁移
- test：按测试环境策略控制
- production：默认不隐式自动迁移，需显式开启或独立执行

这样可以避免将来在生产环境中因为启动即迁移带来不可控风险。

## Phase 2 交付物
- 新的配置结构
- `.env.example`
- 启动装配后的 `main.go` 及相关 bootstrap 代码
- 启动与运行模式说明文档

## Phase 2 风险与缓解
### 风险 1：旧脚本依赖现有相对路径
缓解：逐个检查启动和部署脚本，统一迁移到配置路径。

### 风险 2：本地开发习惯受影响
缓解：保留开发环境最小便利，但通过 `.env.example` 和文档降低切换成本。

### 风险 3：自动迁移策略调整引发环境差异
缓解：在设计中先保留兼容开关，待后续阶段再进一步独立迁移入口。

## Phase 2 验收标准
- 启动流程职责清晰，`main.go` 不再承担过多实现细节
- 关键配置缺失时可以快速定位问题
- 开发 / 测试 / 生产模式边界明确
- 运行路径不再依赖隐式相对目录猜测

# 测试与验证策略

本轮虽然不做大规模业务重构，但仍需最小验证保障：

1. 后端配置加载测试更新或补齐
2. 启动流程在开发环境可正常运行
3. 路由与核心业务接口行为不发生预期外改变
4. 前端开发模式与 API 代理路径不被本轮修改破坏
5. `git status` 与目录结构符合新的仓库边界预期

## 建议验证方式
- 配置单测
- 启动冒烟测试
- 关键接口冒烟检查
- 本地前后端联调检查

# 实施顺序

建议严格按以下顺序执行：

1. 盘点现有运行目录与本地产物
2. 收敛 `.gitignore`
3. 编写目录职责文档
4. 盘点现有配置项与路径依赖
5. 重构配置结构
6. 拆分启动流程
7. 调整自动迁移策略
8. 完成验证

# 成功标准

首轮治理完成的标志是：
- 仓库边界清晰
- 运行时目录职责明确
- 启动路径清晰
- 配置结构清晰
- 后续 Phase 3 后端分层重构可以在稳定基础上启动

# 推荐后续动作

当 Phase 1-2 完成后，下一步应进入后端分层重构计划，而不是继续扩散式修修补补。首轮治理的价值，在于为后续大规模结构治理建立稳定地基。如果这一轮做得足够干净，后续 handler/service/repository 分层、前端 feature 化、运维审计治理都会明显顺畅。
